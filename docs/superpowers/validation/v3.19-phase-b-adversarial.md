## Pattern-by-pattern findings

### Pattern 1 — Cross-module state assumptions

**Probed:**

- Grep for `import app` / `from app import` outside `pipeline_runner.py` and tests.
- Grep for direct access to `_file_registry`, `_registry_lock`, `socketio` outside `app.py`.
- Verify which paths run from **worker thread context** (where `__main__ != app`) vs **Flask handler context** (where `import app` works).

**Clean (no new bugs from this pattern):**

- `pipeline_runner.py` correctly resolves the live `app` module via `_app_module()` (uses `sys.modules["__main__"]` fallback to `sys.modules["app"]`). The fix from earlier in v3.19 is structurally sound.
- All `routes/*.py` files use `import app as _app` from Flask handler context, which always succeeds because the Flask harness imports `app.py` as `app`. They never run from a worker thread.
- `stages/` and `engines/` packages never `import app` and never access `current_app`. Decoupled.

**Conclusion:** Pattern 1 surface is clean. The hardening earlier in v3.19 already closed the class.

### Pattern 3 — Per-stage failure handling / observability

**Probed:**

- Existing error file in registry (`90952f0b9f79`): `status=error` with clear message `"v6: asr_primary transcribe profile not found"`. **Error propagation works.**
- Pipeline JSON missing references → fails clearly via `RuntimeError`. Surface OK.

If a job is cancelled during Stage 1 (Qwen3), `cancel_event.is_set()` will be true but `subprocess.run` blocks until subprocess exits or 30 min timeout. ### Pattern 4 — Race conditions on active_kind

**Probed (code reading only — full runtime races skipped due to long V6 stage timing):**

- `_asr_handler` (`app.py:317`) calls `_pipeline_manager.get(f["active_id"])` at JOB pickup time, not at upload time. The `active_id` UUID is snapshot at upload, but the JSON content is re-read. So a `PATCH /api/pipelines/<id>` after upload but before job pickup affects the in-flight job (e.g. changing `qwen3_context` between upload and run). MED — actual user impact only when admin edits during queue backlog.
- `delete_if_owned` on pipeline succeeds even if in-flight jobs reference it. The job worker will then hit `if pipeline is None: raise RuntimeError` and the file ends up `status=error`. Acceptable — but no UI affordance to warn the admin "pipeline has N jobs queued, deleting will fail them."

## Findings (de-duped, sorted by severity)

### Finding B-1: Three `/api/files/<fid>/...` routes always return 400 "file_id required" because `require_file_owner` keys on `file_id` not `fid` [BLOCKER]

**Pattern:** Pattern 2 (URL-arg / decorator drift)
**Surface affected:** stage rerun, stage segment edit, pipeline_overrides — all V6 stage-level proofreading workflows
**Reproduction:**
```bash
curl -X POST  /api/files/c826ef6bdbb9/stages/4/rerun                 → 400 "file_id required"
curl -X PATCH /api/files/c826ef6bdbb9/stages/4/segments/0 -d '{"text":"x"}' → 400 "file_id required"
curl -X POST  /api/files/c826ef6bdbb9/pipeline_overrides -d '{}'     → 400 "file_id required"
```
**Expected:** Each route checks ownership, then runs its handler.
**Actual:** `auth/decorators.py:53 require_file_owner` does `file_id = kwargs.get("file_id")`. The 3 routes above use URL placeholder `<fid>` (not `<file_id>`), so kwargs has `fid=...` but not `file_id=`. Decorator sees `None`, returns 400 before handler runs.
**Test case:** `tests/test_v3_19_phase_b_findings.py::test_finding_b1_stages_routes_dead`
**Suggested fix direction (do not apply):** Either rename `<fid>` → `<file_id>` in the 3 route declarations, OR teach `require_file_owner` to also look for `kwargs.get("fid")`. The first is mechanically simpler; the second is more defensive.

### Finding B-2: `GET /api/pipelines` returns 500 for any non-admin user — `PipelineManager._asr_manager is None` [BLOCKER]

**Pattern:** Pattern 5 (ownership) / Pattern 1 (constructor wiring drift)
**Surface affected:** non-admin pipeline list — every dashboard load for non-admin users fails to populate the pipeline strip

### Finding B-3: `POST /api/files/<id>/translations/approve-all` is a no-op for V6 files — checks wrong field [HIGH]

**Surface affected:** All V6 file proofreading — the "approve all" button is dead

### Finding B-4: `GET /api/files/<id>/subtitle.{srt,vtt,txt}` returns empty body for V6 files — iterates empty `segments[]` [HIGH]

**Pattern:** Pattern 2 (field shape drift)
**Surface affected:** All V6 file subtitle exports (download SRT/VTT/TXT from proofread page)
**Reproduction:**
```
curl /api/files/c826ef6bdbb9/subtitle.srt → empty body
curl /api/files/c826ef6bdbb9/subtitle.vtt → "WEBVTT\n" only
curl /api/files/c826ef6bdbb9/subtitle.txt → empty
# Adding ?source=zh / ?source=en / ?source=bilingual: still empty

V6 files have `segments: []` (V6 stores everything in `translations[]`). The unified list is empty; export emits empty file. ### Finding B-5: `PATCH /api/files/<id>/translations/<idx>` silently writes to wrong field for V6 — `by_lang.zh.text` is never updated [HIGH]

**Pattern:** Pattern 2 (field shape drift) — DATA INTEGRITY
**Surface affected:** Every V6 inline edit — user edits silently miss the canonical store
**Reproduction:**
```
Before: by_lang.zh.text = "高蘭布連卡", zh_text = ""
PATCH zh_text="USER-EDIT"
After:  by_lang.zh.text = "高蘭布連卡" (unchanged!), zh_text = "USER-EDIT"
```
**Expected:** Either (a) `by_lang.zh.text` is updated to the new value, or (b) backend rejects PATCH on V6 with an error directing the user at a V6-aware endpoint.
**Actual:** Endpoint (`app.py:2492-2522`) blindly sets `zh_text`, `status`, `flags`, `baseline_target`, `applied_terms` at top level. `by_lang` is untouched. Downstream consumers that read `by_lang.zh.text` (e.g., a future export path, the V6 web UI) will see stale data; ones that read `zh_text` (renderer, current subtitle export) will see the edit. **Dual-state divergence — depending on which reader, the user sees different "current text" for the same segment.**
**Test case:** `tests/test_v3_19_phase_b_findings.py::test_finding_b5_patch_misses_by_lang_v6`
**Suggested fix direction:** When `entry.active_kind == 'pipeline_v6'`, PATCH must also mirror the new text into `by_lang.<first_lang>.text` (or take the target lang from a request field).

### Finding B-6: V6 `POST /api/render` with `subtitle_source=zh` produces silent video for 81/82 segments — renderer reads `zh_text` not `by_lang.zh.text` [HIGH]

**Pattern:** Pattern 6 (render fidelity) + same root as B-5
**Surface affected:** V6 ZH render — the primary V6 output path
**Reproduction:** After approving all 82 V6 translations and rendering source=zh, only the 1 segment with a manually-PATCHed `zh_text` shows subtitles; the other 81 are black. `warning_missing_zh: 81` is reported but render still produces a "valid" 147 s MP4 that misleads the user.
**Expected:** Renderer reads V6 canonical `by_lang.<lang>.text` for V6 files (and emits the 82 cantonese subtitles into the burn).
**Actual:** `app.py:2832` builds `translations` from registry and passes to `_subtitle_renderer.generate_ass(...)`, which delegates to `subtitle_text.resolve_segment_text` which reads `seg.get("zh_text")`. No V6 / `by_lang` awareness anywhere in the render pipeline.
**Test case:** `tests/test_v3_19_phase_b_findings.py::test_finding_b6_render_source_zh_empty_v6`
**Suggested fix direction:** Same as B-5. Renderer + `subtitle_text.resolve_segment_text` need a V6 branch that picks text from `by_lang.<lang>.text`.

### Finding B-7: V6 `POST /api/render` with `subtitle_source=en` for a Cantonese-source V6 file burns nonsense — falls back to raw Qwen3 dump [HIGH]

**Pattern:** Pattern 6 (render fidelity / fallback semantics)
**Surface affected:** V6 zh-content render where user mistakenly picks source=en
**Reproduction:** Cantonese V6 file `c826ef6bdbb9` rendered with `subtitle_source=en` proceeds without warning (the `subtitle_source=en` branch at `app.py:2814` BYPASSES the approval gate). Renderer falls back to `t.get('source_text')` which for V6 is the Qwen3 dump (`"HIGHLANDBLINKisa"`, `"specialhorsetomeIdon'tthink"` — no whitespace). Result: 147 s MP4 with whitespace-stripped pseudo-EN gibberish burned in. No `warning_missing_en` is emitted to signal the data is wrong.
**Expected:** Either (a) reject `subtitle_source=en` for V6 files whose `source_lang != "en"`, or (b) emit `warning_missing_en` so frontend can warn the user.
**Actual:** Proceeds silently. The 1 PATCHed segment burns "PROBE-EDIT-VALUE-X9" instead — partial-render-success in a way that masks the underlying mismatch.
**Test case:** `tests/test_v3_19_phase_b_findings.py::test_finding_b7_render_source_en_for_zh_v6`
**Suggested fix direction:** Add `source_lang` awareness — if the V6 pipeline's `source_lang != "en"` AND the requested render `subtitle_source == "en"`, return 400 with a clear message.

### Finding B-8: V6 Qwen3 subprocess does not honor `cancel_event` — cancel can hang for full subprocess runtime [HIGH]

**Pattern:** Pattern 3 (cancel + observability)
**Surface affected:** Any in-flight V6 job cancellation during Stage 1 (qwen3_per_region)
**Reproduction:** Code reading. `engines/transcribe/qwen3_vad_engine.py:109` calls `subprocess.run(..., timeout=1800)`. No `cancel_event.is_set()` check; no `Popen` + poll loop; no `proc.terminate()` on cancel. If the worker thread receives a cancel during Stage 1, the cancel sits in `_check_cancel` ahead of the next stage, which won't run until Stage 1 returns (up to 30 min worst case).
**Expected:** Either (a) Switch to `subprocess.Popen` + polling loop that checks `cancel_event` every N seconds and terminates the subprocess, or (b) document that V6 cancel has stage-granularity.
**Actual:** Subprocess silently runs to completion (or 30 min timeout) before cancel takes effect. **Subprocess leak risk:** if backend is killed while subprocess is running, the py3.11 child is orphaned (no reaper). Verified by reading code; not reproduced in a live run because no current Qwen3 jobs were running.
**Test case:** `tests/test_v3_19_phase_b_findings.py::test_finding_b8_qwen3_subprocess_no_cancel`
**Suggested fix direction:** Replace `subprocess.run` with `Popen` + a polling loop; on `cancel_event.is_set()`, send `terminate()` then `kill()` after 5 s grace.

### Finding B-10: V6 pipeline JSON re-read at job pickup, not at upload — admin PATCH during queue affects in-flight runs [LOW]

**Pattern:** Pattern 4 (race conditions on active_kind)

## Acceptance gate

**Severity tally:** 2 BLOCKER, 6 HIGH, 1 MED, 1 LOW = 10 findings.

Phase B finds **2 BLOCKERS**, both newly discovered (not duplicates of Phase A's 1 BLOCKER):
- **B-1**: 3 dead routes due to decorator/URL-arg name mismatch — V6 stage rerun + stage segment edit + pipeline_overrides are 100% non-functional for everyone.
- **B-2**: GET /api/pipelines 500s for any non-admin user — pipeline strip empty on non-admin dashboards.

**Verdict on V6 graft structural soundness:**

The V6 *transcription pipeline itself* (VAD → Qwen3 → MLX → Merge → Refiner) runs and produces correctly-shaped data in the registry (Phase A confirmed). The V6 graft into the existing Profile-shaped API surface (`/api/files/...` endpoints) has at least **7 field-shape drift bugs** (B-3, B-4, B-5, B-6, B-7, B-9, and Phase A Findings 1+2). Most of these are cohesive — they share the same root cause: legacy endpoints assume `t.get('status')` / `t.get('zh_text')` / `entry.get('segments')` and were never updated to also read from `by_lang` / `entry.translations[]` for V6 mode.

**Combined with Phase A:** Phase A (3 bugs: 1 BLOCKER + 2 HIGH) + Phase B (10 bugs: 2 BLOCKER + 6 HIGH + 1 MED + 1 LOW) = **13 distinct bugs across the V6 graft**, with **3 BLOCKERs** total. Most concerning is that the V6 file is currently not usable end-to-end via any UI path: the proofread page shows zero segments (Phase A Finding 1), approve-all is dead (B-3), inline PATCHes silently miss the canonical store (B-5), subtitle exports are empty (B-4), and render produces silent video (B-6). Of the 6 V6 surface paths a user might touch, **0 work correctly** today.

**Recommendation:** Do NOT merge `dev` to `main` until at least B-1, B-2, B-3, B-4, B-5, B-6 (the cohesive V6 graft cluster) are fixed. A single coordinated patch can close all 6 — they share the same fix shape (add V6 branches that read/write `by_lang` and `translations[]`).
